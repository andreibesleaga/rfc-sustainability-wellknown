# Adapter examples: originals → transformed

For every source this gateway ships an adapter for, this directory carries a matched pair:

| Directory | Contents |
|---|---|
| `originals/` | A real (or realistic, sourced) upstream response/input exactly as that producer emits it — vendor field names, vendor units, vendor shape. Not a draft document. |
| `transformed/` | The exact `/.well-known/sustainability` JSON document this repository's TypeScript adapter + `Publisher` derive from the corresponding original — generated by **running the real code**, not hand-written. |

Every vendor shape below was checked against that vendor's own published API documentation
(sources cited per row); corrections found during that check (see "Verification notes")
were applied to the fixtures, not glossed over.

Regenerate everything at any time with:

```bash
cd publisher
npm run build
node examples/generate-examples.js
```

Every file in `transformed/` passes **both** independent validators
(`schemas-validators/validator-json.py` — Python JTD, and `validator-cddl.py` — Ruby
CDDL); see the repo's `validate-all.sh` / CI for the automated check. `updated` timestamps
in the transformed files reflect the actual moment they were generated (they are not
meant to be re-frozen; regenerate them, don't hand-edit them).

## Verification notes (read before trusting a shape blindly)

Every vendor shape here was checked against that vendor's own docs; two are not
literal 1:1 matches to a real endpoint, and that's stated plainly rather than hidden:

- **Kepler version.** `kepler_node_platform_joules_total` was Kepler's real exported
  metric pre-0.10 (confirmed: https://sustainable-computing.io/archive/design/metrics/).
  Kepler's June 2026 "re-architected" release (CNCF blog:
  https://www.cncf.io/blog/2026/06/30/kepler-re-architected-improved-power-accuracy-and-a-community-call-to-action/)
  replaced it with the `kepler_platform_watts` gauge and per-domain
  `kepler_node_{cpu,gpu}_joules_total` counters
  (https://github.com/sustainable-computing-io/kepler/blob/main/docs/user/metrics.md).
  This example models a **legacy (pre-0.10) Kepler deployment**; the adapter's
  `query` option accepts any PromQL, so pointing it at the current metric names works
  identically against a modern Kepler.
- **Microsoft Sustainability Manager energy field.** The confirmed real
  `Microsoft365Emission` OData entity carries no energy-consumption field at all — only
  `totalEmissions` per tenant/region/scope record. This example therefore supplies energy
  via the adapter's `energyKwh` fallback (representing a separate metering/billing
  export), exactly as a real integration would need to, rather than inventing an
  `energyKwh` field on the vendor entity that doesn't exist.
- **Watershed.** Watershed's public API (https://api-docs.watershed.com) is a job/export
  management API, not a direct footprint-query API: `GET /reference/getfootprint` returns
  metadata only (id, name, timestamps — no emissions numbers), and
  `GET /reference/getcomprehensivefootprintexport` returns an async export job whose
  actual Scope 1/2/3/energy numbers are delivered as a downloaded CSV/Parquet file, not
  inline JSON. The flat shape used here (`totalEmissionsKgCo2e`, `scope1Kg`, …) is an
  **illustrative simplification** representing the common pattern of a small internal
  service that downloads and flattens that export into JSON for internal consumption —
  it is not Watershed's literal public endpoint response.

Everything else in the table above (Climatiq, CO2.js, Greencheck, carbon.txt hosted API,
Salesforce field names) was checked field-by-field against the vendor's own current
documentation and matches.

## Reading the transformations

A few worth pointing at directly, since they show the normalization logic rather than a
trivial field rename:

- **Climatiq** (`co2e_unit: "kg"`) → `carbon-footprint` is in **gCO2e** (`690 kg × 1000 =
  690000`); the adapter fails loudly on any `co2e_unit` it doesn't recognize rather than
  silently assuming grams. The vendor's `emission_factor.id`/`activity_id` split is
  intentionally not copied through (provenance belongs behind `methodology-uri`).
- **Kepler** sums two nodes' joule counters (`486000000000 + 522000000000 =
  1.008 × 10¹² J`), converts to kWh (`÷ 3,600,000 = 280000`), then multiplies by the
  configured grid intensity (`× 230 = 64400000 g`).
- **Salesforce** and **Watershed**: `scope-1 + scope-2 + scope-3` sums exactly to
  `carbon-footprint` in both outputs — the adapters pass the vendor's own scope breakdown
  through unchanged, and the vendor totals already reconcile.
- **Microsoft Sustainability**: two OData pages are walked via the cursor
  (`@odata.nextLink`/`$skiptoken`) and their per-record `totalEmissions` are summed into
  one period total (`1.82 + 0.41 + 0.95 = 3.18`); the tenant's cloud-usage emissions are
  classified as **Scope 3** (`scope-3` = total) from the customer's perspective, per the
  adapter's design.
- **carbon.txt / CO2.js**: `energy × carbon-intensity-gCO2-per-kWh ≈ carbon-footprint` by
  construction — the adapters recover CO2.js's intensity-independent operational energy
  and re-apply the *reported* intensity, so the two numbers are always self-consistent.

## Not included here

Adapter *upstream-API* test fixtures used by the automated test suite live in
`publisher/test/fixtures/` — they exist to make tests deterministic and are not curated
for this "originals vs. transformed" narrative (though several are close relatives of the
same recordings, now with the vendor-shape corrections above applied here first). This
directory is the reader-facing, documented, source-verified version of the same idea.
